- **Clutch lining picker + label overlay** (`ClutchConfigurator` section, placed below Hero on homepage): Rendering technique changed from canvas grayscale+multiply-blend to layered image-swap. Reason: canvas blend-mode can only handle hex colors; real fabric patterns (zebra, floral, denim, etc.) need a full image, so the approach changed to pre-composited PNGs. Each lining option is now a 1010×1024 PNG with the lining-shaped area filled with the fabric pattern/color, transparent outside — just drawn on top of the base photo at (0,0), no offset math needed. Implementation is three `<Image fill>` layers inside a `relative w-full aspect-[1010/1024]` container: base photo → lining PNG → label. No canvas, no pixel math, no refs/effects — just `useState` for the selected option. Hebrew label "צבע בטנה", RTL select. Body color picking still deferred. `lining-mask.png` retained in `public/` for reference/future mask generation but no longer used for rendering.
  - **Label overlay constants — unchanged, browser-calibrated**: `LABEL_CENTER_X=0.45`, `LABEL_CENTER_Y=0.36`, `LABEL_WIDTH=0.11`, `LABEL_HEIGHT=0.06`, `LABEL_ROTATION=-0.15` rad (≈−8.6°). Expressed as fractions of 1010×1024. Do not recalculate from the Python bbox derivation — these final values supersede it.
  - **Lining options with real PNG assets**: `זברה` (`lining-zebra.png`) ✓, `אריות` (`lining-lion.png`) ✓.   - **Follow-up fix — header height regression**: the swap initially pushed the header taller. Root cause: Tailwind Preflight's `img, video { height: auto }` overrode the HTML `height` attribute Next's `Image` sets, so the browser recomputed height from the SVG's declared natural ratio (a square `2000×2000`) instead of the intended `164×35` box. Fixed by forcing `style={{ height, width, objectFit: "contain" }}` inline on the `Image` in `Logo.tsx` (inline style wins over the Preflight class rule).
  - **Follow-up fix — SVG cropped to its content**: measured the actual glyph paths inside `logo-header.svg` (bounding box x:[37,1470] y:[563,851] within the declared `1500×1500` viewBox) — the letterforms occupied only ~19% of the canvas height, huge transparent padding baked into the file, which is what made the wordmark render tiny once height was constrained. Fixed by editing only the root `<svg>`'s `viewBox` (→ `0 550 1500 320`) and `height` (→ `426.666667`, same scale factor) attributes — no path data touched. Verified by rasterizing with `sharp` (Next's already-installed image dependency, no new package needed): wordmark now fills the frame edge-to-edge with only small even margins, no clipped letterforms, transparency (RGBA) intact.
- **Signature stitch refined — straight dash rhythm + animated needle** (`StitchLine.tsx`, Hero only for the needle): built with the `frontend-design` skill. The line changed from an irregular wavy multi-curve path to a single straight line with an even `9 6` dash rhythm (round caps) — the "handmade" quality now comes from the stitch rhythm and draw-in, not from wobble. The animated (Hero) variant's mechanism changed from a CSS `clip-path` reveal to an SVG `<mask>` built from a classic solid stroke-dashoffset "draw" line (`useId()`-scoped mask id, safe for multiple instances) — this keeps the visible dash pattern fixed while the mask sweeps it into view right-to-left, so individual stitches appear to snap into place rather than the whole line just fading/wiping in. A minimal filled needle silhouette (ink color, point leading, small eye near the trailing/blunt end — deliberately a filled shape rather than a stroked outline, since a stroked version tested first read as a leaf/eye icon rather than a needle) travels the same path via CSS `translateX`, timed to arrive exactly as the thread finishes, then does a brief scale `1.05 → 1` "landing" settle (~150ms) and stops permanently — no loop, no idle animation. `prefers-reduced-motion` skips straight to the fully-drawn end state (thread complete, needle resting at the tip). `docs/DESIGN_SYSTEM.md`'s "Signature Element — The Stitch" section updated to match. Static dividers (`ProductCarousel`, `ClutchConfigurator`) only picked up the straight-line/dash-rhythm change — no needle, no mask, unchanged otherwise.
  - **Verification**: `tsc --noEmit` passes. Design was iterated visually by rasterizing candidate SVG markup with `sharp` at each stage (dash rhythm, needle silhouette shape/scale, final resting-state composition) before wiring up the CSS animation, since no browser/screenshot tool was available in this environment — the animation's live motion (mask sweep timing, needle sync, settle bounce) was not confirmed in an actual browser. Recommend a quick look at the Hero on load, both at 375px and desktop, to confirm the timing feels right.
  - **Follow-up — slower timing + needle rides above the thread, then plunges to land**: durations doubled (thread reveal `1.1s → 2.2s`, needle total `1.25s → 2.5s`, same 88/94/100% breakpoints so the sync is preserved exactly) with a gentler `cubic-bezier(0.16, 1, 0.3, 1)` glide. The animated variant's `viewBox` grew from `"0 0 320 8"` to `"0 0 320 20"` (container `h-2 → h-5`, static dividers untouched) and the thread moved down to `y=16`, so the needle — left authored at its original `y≈4` coordinates, unchanged — now automatically rides ~12 units above the line for the whole glide instead of sitting on top of it. In the last 12% of the timeline it drops (`translateY(0) → translateY(12px)`) with its own snappier per-keyframe easing (quick ease-in drop, `cubic-bezier(0.34, 1.56, 0.64, 1)` overshoot on the settle) so it reads as a decisive plunge landing on the fabric, not a continuation of the slow glide.
  - **Bug found and fixed — stitch rendered far wider than intended**: this environment does have a headless Chrome (`C:\Program Files\Google\Chrome\Application\chrome.exe`), so this pass was actually verified in a real browser via `--headless --screenshot` (no Playwright/Puppeteer installed, so driven directly). That surfaced a real, pre-existing bug: the animated variant's own base classes included `w-full` alongside Hero's `className="w-28 ..."` override — both target `width` at equal Tailwind specificity, and `w-full` was winning, so the Hero stitch rendered nearly full-column-width instead of the intended compact `w-28`. This predates this session (same conflict existed with the original wavy line, just never caught — `PROGRESS.md`'s own earlier note flagged that pass as visually unverified). Fixed by dropping `w-full` from the animated branch's className only (static dividers still need it, unchanged); Hero's `w-28` is now the sole width driver. Confirmed via screenshot: correct compact width, and with `--force-prefers-reduced-motion`, the thread renders fully drawn with the needle resting exactly on the line at the tip, as designed.
