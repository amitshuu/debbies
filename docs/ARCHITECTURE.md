# Debbies ‚Äî Architecture & Conventions

## Folder Structure

```
src/
  app/
    (storefront)/
      page.tsx                 ‚Üí homepage
      shop/
        page.tsx                ‚Üí product listing
        [slug]/page.tsx          ‚Üí single ready-made product
      clutch/
        page.tsx                ‚Üí custom clutch configurator
      cart/
        page.tsx
    api/
      stripe/webhook/route.ts   ‚Üí only place a raw API route is justified pre-phase-3
    layout.tsx
    globals.css
  components/
    ui/                         ‚Üí generic, reusable, no business logic (Button, Card, Badge)
    storefront/                 ‚Üí feature-specific (ProductCard, FabricPicker, CartDrawer)
    layout/                     ‚Üí Header, Footer, MobileNav
  lib/
    prisma.ts                   ‚Üí Prisma client singleton
    stripe.ts                   ‚Üí Stripe client singleton
    supabase.ts                 ‚Üí Supabase client singleton
    validations/                ‚Üí zod schemas
    utils.ts
  store/
    cart-store.ts                ‚Üí Zustand store
  types/
    index.ts                    ‚Üí shared TS types not generated by Prisma
prisma/
  schema.prisma
  migrations/
```

## Naming Conventions

- Components: `PascalCase`, named exports preferred over default exports (easier to grep/refactor), e.g. `export function ProductCard()`.
- Files: match the component name ‚Äî `ProductCard.tsx`, not `product-card.tsx`, for component files. Utility/lib files use `kebab-case.ts`.
- Component names describe what they render, not how: `FabricPicker`, not `Step2Selector`.
- Server actions: verb-first, e.g. `createOrder`, `getProductBySlug`.

## Component Rules

- One component, one responsibility. If a component's JSX needs a comment to explain a section, split it.
- No business logic inside `components/ui/*` ‚Äî those are pure presentation.
- Data fetching happens in Server Components by default. Only mark `"use client"` when the component genuinely needs interactivity (state, effects, event handlers, the fabric/lining visualizer, etc).
- Avoid prop drilling more than 2 levels ‚Äî reach for the Zustand store or composition instead.

## Data Fetching Rules

- No client-side `useEffect` + `fetch` for data that could be fetched server-side. Default to Server Components fetching directly via Prisma.
- Only fetch what's needed for the page being rendered ‚Äî no "fetch everything and filter client-side" patterns.
- Mutations (add to cart is client-only/Zustand; checkout, order creation) go through Server Actions, validated with zod, per SECURITY.md.

## Styling Rules

- Tailwind utility classes directly in JSX ‚Äî no separate CSS files per component unless there's a genuine reason (complex animation, etc).
- Design tokens (colors, fonts, spacing scale) defined once in `tailwind.config.ts`, referenced by name everywhere ‚Äî never hardcoded hex values in components. See DESIGN_SYSTEM.md once finalized.
- Mobile-first: write the unprefixed (mobile) styles first, then layer `sm:` / `md:` / `lg:` breakpoints on top.

## Code Style

- No comments explaining *what* code does ‚Äî code should be self-explanatory through naming. A comment is acceptable only to explain *why* something non-obvious was done a certain way (e.g. a workaround for a library quirk).
- No unnecessary abstraction ‚Äî don't build a generic "configurable field system" for two dropdowns. Solve the problem in front of us; generalize when a second real use case appears.
- Strict TypeScript ‚Äî no `any`, no `@ts-ignore` without a one-line justification.
